## 0. 🎯 Главная дисциплина релиза (проект «забетонирован»)
Игра — **релизная, в открытом доступе**. Никаких случайных правок.
- **Меняй ТОЛЬКО нужный блок.** Перед правкой найди блок в `docs/ARCHITECTURE.md` и трогай **только его**.
  Просят поменять меню — меняешь меню, а не физику/рендер/слияние. Соседние блоки не трогать «раз уж открыл файл».
- **Без костылей:** чинить источник проблемы, минимальной целевой правкой.
- **Каждое изменение → запись в `docs/CHANGELOG.md`** (что, зачем, файлы/блоки, версия). Это обязательно.
- Замороженные зоны (`physics.js`, ядро физики/цикла, библиотеки) — не трогать без явной просьбы владельца.

## 3. 🔴 Git-безопасность (ГЛАВНОЕ — однажды из-за нарушения потеряли большой пласт работы)
- **НИКОГДА** не делать `git reset --hard`, `git checkout -- <файл>`, `git stash drop`, `git push --force`,
  если в рабочей папке **могут быть незакоммиченные правки**. Сначала — `git status`.
- Если есть незакоммиченные изменения — **сначала закоммитить** (или сохранить в отдельную ветку), и только потом синхронизироваться.
- Перед любой сборкой — `git status` и `git commit`, чтобы работа не пропала.
- Не удалять резервные ветки/теги (`recovered-wip`, `archive/*`).

## 4. 🔢 Версии для Google Play (чтобы Play не ругался «такая версия уже есть»)
Файл версий: `android/app/build.gradle` → `versionCode` и `versionName`.
- **`versionCode`** — целое число, Google Play требует, чтобы оно **строго росло** с каждой загрузкой.
  Загрузка числа ≤ уже загруженного → Play отклоняет.
- **`versionName`** — человеческая версия (строка, напр. `"2.6"`), тоже увеличиваем.

**ПРАВИЛО:** перед каждой сборкой для Play — **увеличить `versionCode` на 1**, обновить `versionName`,
и **обновить строки ниже в этом файле**. Число **никогда не переиспользовать и не уменьшать**.

## 5. Сборка (для Gemini)
- Собирать **подписанный `.aab`** из этой папки (`D:\мои игры`), ветка `main`.
- Ключ подписи — из защищённого места (`~/.catdrop/keystore.properties` или переменные
  `CAT_DROP_KEYSTORE`, `CAT_DROP_STORE_PASSWORD`, `CAT_DROP_KEY_ALIAS`, `CAT_DROP_KEY_PASSWORD`).
  Ключей в репозитории НЕТ и быть не должно.
- Если файлы уже актуальны (Claude подготовил и закоммитил) — **`git pull` не нужен, просто собрать**.
- Обязательно поднять versionCode (см. п.4).

## 6. Структура проекта
- `www/` — сама игра (PWA): `js/game.js` (движок Matter.js), `js/physics.js`, `js/state.js`,
  `js/diagnostics*.js` + `js/sentry.min.js` (диагностика), `js/cloud_save.js`, `sw.js`.
- `android/` — обёртка Capacitor под Android.
- `store-listing/` — картинки для витрины магазина (**НЕ входят в APK**).
- `docs/` — рабочие документы (`monitoring-catalog.md`, архив идей).
- **Не трогать без отдельной просьбы:** `js/physics.js` (замороженный эталон физики).

## 7. Service Worker (кэш)
Игра — PWA, кэш cache-first. **При любом изменении файлов в `www/`** — поднять `CACHE_NAME` в
`www/sw.js` (иначе апдейт не долетит до установленной версии) и бампнуть `?v=N` у изменённых скриптов в `index.html`.

## 8. Диагностика (Sentry «чёрный ящик»)
- Проект Sentry `cat-drop` (регион DE). Код: `www/js/diagnostics.js` (кадры/зависания/паспорт устройства),
  `www/js/diagnostics_watchdogs.js` (физ. аномалии). План — `docs/monitoring-catalog.md`.
- Данные обезличенные; декларация «Безопасность данных» в Play Console заполнена.

## 9. Очистка папки
- **`npm run clean`** — удаляет старые сборки (`*.apk/*.aab`), кэш Android-сборки, `tmp/`. Запускать при захламлении.
- `.gitignore` держит сборки/кэш/ключи вне git.

## 10. Дисциплина качества
- Перед правкой бага — прочитать этот файл и найти **источник** поведения, а не лепить костыль.
- После правки — **проверить на реальном сценарии** (симуляция Matter.js и/или страница в браузере),
  чтобы тест ловил исходный риск и проходил после фикса. Не считать готовым без проверки.

## 11. Технические ограничения
- Реальная сборка `.apk`/`.aab` — на ПК владельца (Windows). Игра должна работать офлайн, поэтому
  сторонние скрипты (Sentry и т.п.) хостятся **локально**, не с CDN.
